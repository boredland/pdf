## Automatic language detection
- [ ] Detect the page language(s) before OCR runs so users don't have to
      pick. Two approaches worth prototyping:
        a) Run Tesseract OSD first (already loaded for rotation) — it
           returns `script` + `script_confidence`. Map Latin/Cyrillic/
           Arabic/CJK/etc. to a default language code, let users confirm.
        b) Kreuzberg-style: run a tiny LSTM on a sampled strip of each
           page, vote across pages, pick the traineddata. Offline-safe
           once the model ships (~1 MB).
- [~] Once detection is in, auto-suggest a matching language in the
      pill and let a one-click action download + select it. Tracked
      below.

## Output compression
- [x] Builder: embed mask + background separately (was: lossless PNG of
      the composed image, ~18× source size) → ~7-8× after mask+bg split
      and 0.75-scale mask. Further drops blocked on JBIG2.  (shipped)
- [ ] Real JBIG2 encoder for the mask layer — **highest-impact remaining
      compression item**. Our "optimized mask PNG" is 8-bit RGBA with
      all pixels clamped to 0/255 and flate-compressed by the Canvas
      PNG encoder; a real 1-bit-per-pixel encoding via jbig2enc gets
      another 5-10× reduction on text-heavy pages. jbig2enc-wasm exists
      but builds are flaky — vendor our own if needed.
- [ ] Real JPEG 2000 encoder for the background layer (current output is
      JPEG via Canvas). openjpeg-wasm is the candidate.
- [x] PDF native /ImageMask for the mask layer (1 bit per pixel +
      flate instead of 32-bpp alpha PNG). Drops skew.pdf output from
      7.6× to 2.0× of source.                             (shipped)
- [x] MRC worker measures mask coverage and sets skipMask=true when
      it's <0.5% (blank) or >35% (photo-dominated). Builder then
      omits the /ImageMask overlay for those pages.         (shipped)
- [ ] cardinal.pdf 77 KB → 674 KB (8.8×) is still bad — mask coverage
      lands in the valid text-heavy band (~7.5%) so skipMask doesn't
      fire, but the bg layer alone (JPEG 0.5 quality, 0.5 scale)
      overshoots the source because the source PDF already embeds
      high-quality JPEGs we re-decode + re-encode at lower settings.
      Better fix: extract the original image XObjects from the source
      PDF and embed them directly when they're present, bypassing MRC
      for photo-heavy pages. Requires mupdf image extraction.
- [ ] Investigate zstd PDF stream filtering (PDF 2.0 feature) for
      non-image streams — mupdf support TBD.
